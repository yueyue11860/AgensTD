目前你的电脑卡死，大概率是 AI 编程助手在生成代码时，埋下了以下三个“地雷”之一。我们需要通过控制变量法来排查。

第一步测试： 先把前端 React 关掉，只启动 Node.js 后端。如果 CPU 立刻飙到 100%，那就是后端地雷；如果后端很安静，一打开前端网页才卡死，那就是前端地雷。

💣 嫌疑犯 1：后端的“无限跑圈”引发了死循环（后端概率：90%）
我们刚刚引入了怪物走到终点后“重置路标索引”的无限跑圈机制。AI 极有可能在 Enemy.move(deltaTime) 这个方法里写出了一个永远跳不出的 while 循环。

错误重现： 很多移动算法会写成 while (移动距离 > 0) { 走到下一个拐点; 扣除距离; }。如果怪物到达了最后一个点，触发了无限循环重置，但由于浮点数精度问题，或者两点距离被算成了 0，这个 while 循环就会在 1 个 Tick 的瞬间执行几百万次，直接把 Node.js 的单线程彻底锁死。

排查方法： 检查后端的 Enemy 类里的移动计算逻辑，特别是当 targetWaypointIndex 重置时的代码，确保没有任何可能导致无限循环的 while 或递归调用。最安全的做法是在 move 里加上保护机制：每个 Tick 最多只允许跨越 1 个节点。

💣 嫌疑犯 2：错误的 setInterval 调用（后端概率：10%）
这是一个非常经典的 JavaScript 语法坑。

错误写法： setInterval(this.tick(), 100); （注意 tick 后面带了括号）。这会导致引擎在启动的瞬间，同步、无限次地递归调用 tick 函数，直到栈溢出或 CPU 爆炸。

正确写法： 必须传函数引用，setInterval(() => this.tick(), 100); 或 setInterval(this.tick.bind(this), 100);。

💣 嫌疑犯 3：前端 React 的“渲染风暴”（前端概率：高）
如果后端没卡死，是浏览器卡死了（风扇狂转，网页无响应），那绝对是前端的 React 状态更新写崩了。

错误重现： 后端每 100ms 发送一次 WebSocket 数据。前端在 useEffect 里监听数据并调用 setGameState(data)。如果这个 useEffect 没有写依赖项数组 []，或者在组件渲染的直接过程里触发了状态更新，就会导致：收到数据 -> 渲染 -> 触发更新 -> 重新渲染 -> 陷入死循环。

排查方法： 检查前端的 Socket 监听代码。确保 socket.on('state') 是在组件挂载时只绑定一次（依赖项为空数组 []）。
