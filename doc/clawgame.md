# A2A 硬核 Roguelike 竞赛项目规划书

## 1. 项目目标

本项目面向 A2A 参赛场景，目标是构建一款可由 agent 接入并自动通关挑战的高难度硬核 roguelike 游戏。项目不依赖传统游戏引擎，只使用前端、Supabase 与后端服务实现。画质不是重点，重点是规则深度、状态复杂度、决策链长度、评测公平性与可复现性。

游戏需要同时满足以下目标：

- 适合 agent 接入与自动决策
- 游戏规则足够硬核，通关计算量大
- 单局状态复杂，适合长程规划和构筑博弈
- 支持四档难度逐级解锁
- 地狱难度支持基于通关时间的赛季天梯排名
- 后端权威模拟，保证公平、防作弊、可回放

难度目标通关率如下：

- 简单：50%
- 普通：30%
- 困难：10%
- 地狱：5%

以上通关率以官方基线 agent 评测集为标定对象，而不是对所有未知 agent 做绝对承诺。

## 2. 总体产品定义

游戏类型定义为：

高难度塔防式 roguelike + 路线选择 + 构筑成长 + 主动技能干预 + 多资源管理。

玩家或 agent 在单局中需要完成以下任务：

- 在地图上建设和调整防御体系
- 管理资源与经济节奏
- 在区域节点中选择路线
- 购买或放弃商店物品
- 在事件中做分支抉择
- 处理精英敌人与 Boss 的克制机制
- 在中后期根据构筑形态进行转型

游戏的本质不是拼操作速度，而是拼长期规划、局部搜索、风险控制和构筑协同。

## 3. 技术架构

### 3.1 总体原则

整个系统采用服务端权威模拟架构。

前端只负责展示、调试、观战与回放，不参与真实结算。agent 也不负责计算结果，只负责提交动作意图。所有规则、随机数、状态推进、事件结算、胜负判断全部由后端统一执行。

核心原则如下：

- 服务端权威
- 确定性模拟
- 事件流回放
- 数据驱动规则配置
- 可横向扩容的评测执行架构

### 3.2 系统组成

系统分为四层。

第一层：前端

- 提供玩家调试台
- 提供 agent 对局观察界面
- 提供回放查看器
- 提供排行榜界面
- 提供局内状态可视化和日志可视化

第二层：Supabase

- Auth 用于账号、队伍、agent 身份认证
- Postgres 用于存储业务数据、对局结果、排行榜、解锁进度
- Realtime 用于对局状态更新、排队进度同步
- Storage 用于存储回放文件、日志文件、评测产物
- Edge Functions 用于轻量编排和对局令牌下发

第三层：后端应用层

- API 网关：统一接入前端和 agent 请求
- 对局编排服务：创建 run、校验权限、分配种子、写入数据库
- 排名服务：计算赛季成绩、写入排行榜
- 解锁服务：根据通关记录解锁下一档难度
- 回放服务：生成回放索引、组织事件流和快照

第四层：模拟执行层

- 模拟器服务：真实执行游戏规则
- Worker 集群：并发跑单局模拟
- 调度器：按难度与资源配额分配任务
- 基线 agent 服务：用于平衡性测试和通关率标定

### 3.3 技术选型建议

推荐两种实现路线。

路线 A：全 TypeScript

- 前端：Next.js + React + Canvas 或 PixiJS
- 后端：Node.js + TypeScript
- 模拟执行：Node worker_threads 或独立模拟进程
- 数据访问：Supabase 官方 SDK

优点是开发效率高，整体技术栈统一，适合快速参赛。

路线 B：前后端 TypeScript + 模拟器 Rust

- 前端：Next.js + React
- API 服务：TypeScript
- 模拟器核心：Rust
- 调度与通信：HTTP 或消息队列

优点是性能更强，更适合“单局计算量很大”的目标。

如果以比赛交付速度优先，建议先采用路线 A 做 MVP，再在后续阶段把模拟核心替换为 Rust。

## 4. 游戏核心玩法设计

### 4.1 单局结构

单局由 6 个区域组成。

每个区域包含 5 个普通节点和 1 个 Boss 节点。普通节点从以下类型中生成：

- 战斗
- 精英
- 商店
- 事件
- 营地

单局总战斗数约 30 到 36 场，足以形成长程决策链和构筑演化空间。

### 4.2 地图与战斗模型

- 地图使用规则网格，建议 18×18 或 24×24
- 敌人从多个入口刷新并尝试抵达主堡
- 地图支持动态障碍、地形机关与局部路径重规划
- 玩家可建造基础塔、强化模块、地形装置、辅助设施
- 战斗既包含波次前布置，也包含波次中的有限干预

### 4.3 核心资源系统

建议采用四资源并行设计：

- 金币：主要用于建塔、升级、商店购买
- 热量：高输出构筑的副作用资源，过载会引发惩罚
- 法能：主动技能与法术型塔的释放资源
- 维修点：拆除、改造、重构、修复主堡的关键资源

多资源耦合的目的，是避免策略退化为单一经济最优解，并提升 agent 的计划复杂度。

### 4.4 基础动作集合

agent 或玩家在对局中可执行的动作包括：

- 建造塔
- 升级塔
- 拆除塔
- 安装模块
- 切换攻击优先级
- 释放主动技能
- 使用一次性道具
- 购买商店物品
- 刷新商店
- 选择事件分支
- 在营地进行修整、强化或风险投资

### 4.5 单局核心循环

为了同时兼容人类玩家与 agent，本项目的单局节奏明确拆分为四层：

- 波前准备层
- 波中轻实时干预层
- 波后结算层
- 节点决策层

四层的职责如下。

波前准备层：

- 允许建造、升级、拆除、安装模块、改路、购买波前道具
- 允许查看未来两到三波情报
- 允许有限时暂停，用于完成布局调整
- 是主要的构筑搭建与资源分配阶段

波中轻实时干预层：

- 战斗自动推进，玩家或 agent 只做少量高价值动作
- 允许释放主动技能、切换塔的目标模式、使用一次性战术道具、执行应急维修
- 默认不鼓励大规模重构，不把胜负建立在高频手速上
- 所有波中动作都要有资源代价、冷却或次数限制

波后结算层：

- 结算金币、热量回退、维修恢复、遗物触发、负面状态扩散
- 统计主堡受损、敌人漏怪、塔楼过载、污染扩散等结果
- 为下一波或下一节点生成新的决策约束

节点决策层：

- 发生在商店、事件、营地、Boss 前危机窗口等关键节点
- 允许玩家或 agent 做分支选择、重构路线、投资风险收益项
- 决策有明确时间上限，但时间压力不以操作速度为核心
- 是长期规划和中局转型的主要舞台

这四层共同保证一局游戏既有实时压迫感，也不会退化成纯拼 APM 的传统塔防。

### 4.6 统一动作语义

玩家输入与 agent 接口必须统一到同一套动作语义层。

前端的鼠标点击、QWER 快捷键、按钮点击，都只是动作意图的输入方式；agent 的 HTTP 或 A2A 调用，同样只是在提交相同的动作意图。服务端永远只识别标准化 action，而不识别人类专属或 agent 专属的特殊通道。

建议首版统一动作集如下：

- BUILD：在合法地块建造指定塔核
- UPGRADE：提升塔核等级或激活强化分支
- SELL：拆除塔核并按规则返还资源
- MODULATE：安装、替换或拆除模块
- RETARGET：切换攻击优先级或技能模式
- CAST：释放主动技能
- CONSUME：使用一次性战术道具
- REPAIR：应急维修塔核、地块或主堡部件
- REROUTE：改路、封路、打开支线或部署障碍
- BUY：在商店购买物品、模块、遗物或服务
- REFRESH_SHOP：刷新商店内容
- CHOOSE_OPTION：选择事件、危机窗口、营地或路线分支
- PAUSE_OR_RESUME：进入或退出合法暂停窗口
- NO_OP：显式空动作，用于保留资源或跳过窗口

每个 action 必须定义以下字段：

- action_type
- target_kind
- target_id 或 target_cell
- payload
- observation_version
- issued_at_tick

服务端对每个动作统一执行：

- 合法性校验
- 资源校验
- 冷却校验
- 上下文校验
- 决策窗口校验
- 结果写入动作日志

这样可以保证人类玩家与 agent 在同一规则下竞争，且回放与仲裁始终可复现。

### 4.7 操作方式设计

本项目的人类玩家操作方式明确采用：

鼠标点击负责选择上下文，Q、W、E、R 负责触发当前上下文中的四个高频动作。

该设计的目标不是减少玩法深度，而是把复杂度放在决策与规则理解上，而不是放在记忆大量快捷键和执行连招上。

推荐基础映射如下：

- 鼠标左键：选中塔、地块、敌方威胁目标、商店项或事件选项
- 鼠标右键：取消当前选中或退出次级模式
- Q/W/E/R：触发当前上下文中的四个动作槽
- Space：进入或退出合法暂停窗口
- Tab：切换信息层，例如威胁视图、热量视图、路径视图
- Shift：显示扩展情报，例如预计收益、作用范围、资源预扣结果

Q/W/E/R 不承担菜单翻页，不承担复杂组合技，不承担隐藏命令。它们只负责当前选中上下文中的前四个高价值动作。

例如：

- 选中空地时，Q/W/E/R 可映射为四种可建塔核
- 选中己方塔时，Q/W/E/R 可映射为升级、切换目标模式、应急降温、拆除
- 选中敌方威胁目标时，Q/W/E/R 可映射为标记集火、释放战术技能、查看威胁详情、忽略该目标
- 进入危机窗口时，Q/W/E/R 可直接映射四个决策选项

### 4.8 上下文与动作槽规则

为了避免同一键位在不同场景中的语义漂移过大，必须把操作划分为固定上下文。

首版至少包含四种上下文：

- 空地上下文
- 塔核上下文
- 威胁目标上下文
- 全局危机窗口上下文

空地上下文：

- Q：建造单体输出塔
- W：建造范围塔
- E：建造控制塔
- R：建造功能塔或障碍装置

塔核上下文：

- Q：升级或进阶
- W：切换攻击优先级
- E：执行塔专属主动动作，例如降温、过载、展开、折叠
- R：拆除或重构

威胁目标上下文：

- Q：标记为集火目标
- W：投放单次战术技能
- E：查看反制信息或威胁详情
- R：取消锁定或切回默认策略

全局危机窗口上下文：

- Q/W/E/R：直接对应四个候选决策

动作槽必须有以下展示与反馈：

- 是否可用
- 资源消耗
- 冷却状态
- 禁用原因
- 预计收益或风险摘要

这样玩家只需要理解“当前选中了什么”，就能稳定预期 QWER 的意义。

### 4.9 波中轻实时操作边界

虽然项目允许更自由的实时操作，但波中动作必须严格控制上限。

推荐首版波中只开放以下四类动作：

- 主动技能
- 目标模式切换
- 一次性战术道具
- 应急维修或降温

以下动作默认不在波中自由开放，除非遇到特殊遗物、Boss 机制或高难度规则重写：

- 大规模建造
- 连续拆塔重构
- 无成本路径重绘
- 无限制暂停

这样可以保留战场压力、兼顾观赏性，同时确保玩法核心仍然是路线选择、资源管理、构筑协同与长程规划。

### 4.10 首版核心系统建议

围绕当前规划中的热量、维修点、污染、改路、多入口汇流，MVP 阶段建议优先落以下系统。

塔核首版建议：

- 弩机塔核：单核斩杀，负责精英与重甲处理
- 迫击塔核：范围爆发，负责汇流点清线
- 寒霜塔核：控制延滞，负责冻结链与波次减速
- 诅咒塔核：乘法增伤，负责转化高热或高压目标
- 炉心塔核：高热爆发，负责短时击穿高护甲敌人
- 连锁塔核：依赖布局，负责路口收益最大化

波中主动技能首版建议：

- 过载导通：短时间提高指定塔输出，但快速增加热量
- 紧急降温：立刻降低全局或单塔热量，消耗法能
- 路障投放：短时阻滞指定路径，消耗维修点
- 目标标记：对指定敌群施加集火与易伤效果

敌人反制首版建议：

- 护甲循环敌：压制高频单体塔
- 污染施法者：迫使玩家分配维修点
- 解控先锋：针对减速与冻结流派
- 路径重写 Boss：迫使玩家做中局转型

这些系统足以支撑“规划深、操作轻、压力强”的首版体验。

## 5. 内容体系设计

### 5.1 塔与构筑

建议首版内容池为：

- 18 到 24 种基础塔核
- 60 到 100 种塔改造词缀
- 10 到 15 种主堡模块
- 20 到 30 种一次性战术道具

基础塔可以覆盖如下类别：

- 箭塔：高频单体输出
- 炮塔：范围爆发
- 法塔：异常状态与穿透处理
- 诅咒塔：减速、易伤、标记、资源操控
- 召唤塔：召唤临时单位或障碍
- 热能塔：高伤害高副作用
- 连锁塔：依赖布局触发协同

### 5.2 遗物与事件

建议内容池：

- 100 到 150 个遗物
- 40 到 60 个事件

遗物应强调乘法联动而不是简单加法，例如：

- 每次敌人冻结后，后续爆炸伤害按最大生命值结算
- 每个区域首次拆塔返还资源并提升永久热量上限
- 主堡血量越低，诅咒塔效果越强

事件则用于制造中长期博弈，例如：

- 立即获得强力遗物，但下三区域精英词缀数加一
- 牺牲当前维修点，换取路线预知能力
- 永久增加商店折扣，但所有 Boss 进入第二阶段时获得护盾

### 5.3 敌人与 Boss

建议内容池：

- 80 到 120 种敌人原型
- 20 到 30 个精英模板
- 8 到 12 个 Boss

敌人设计不能只是血量和攻击叠数值，而要具有明确反构筑意图，例如：

- 对高攻速流派免疫部分 hit 触发
- 对召唤流拥有清场能力
- 对减速流具有阶段性解控
- 对单核高伤流具备分裂转移机制
- 对范围爆发流拥有护甲循环和虚体阶段

Boss 应具备阶段切换与规则重写能力，例如：

- 切换路径拓扑
- 暂时封锁建造格
- 复制最近击杀的敌人模板
- 将部分塔临时污染为负面建筑

## 6. 为什么这套游戏适合 agent

这套设计对 agent 友好，原因在于：

- 状态是结构化的，容易以 JSON 观察形式暴露
- 动作空间可离散化
- 既有战术决策，也有战略规划
- 既有短期最优陷阱，也有长期收益权衡
- 局中随机性存在，但可被种子固定和回放复现

这使得参赛 agent 可以采用多种方法：

- 规则驱动
- 启发式搜索
- Beam Search
- MCTS 变体
- 基于模型的规划
- LLM + 工具调用式决策

## 7. agent 接入方案

### 7.1 接口原则

agent 只能提交动作意图，不能直接改写状态。

建议定义以下基础接口：

- 创建 run
- 获取当前 observation
- 提交 action
- 查询 run 状态
- 获取结果与回放

### 7.2 observation 结构

建议 observation 包含以下字段：

- run_id
- tick
- difficulty
- seed
- 区域编号与节点编号
- 地图状态
- 当前资源状态
- 已建塔列表
- 敌人摘要信息
- 主堡状态
- 当前 Buff 与 Debuff
- 商店或事件选项
- 可执行动作列表
- 当前动作时限
- 当前版本号

### 7.3 action 结构

action 只描述意图，建议字段包括：

- action_type
- target_cell
- target_entity
- item_id
- level_choice
- option_index
- observation_version

后端对 action 做以下处理：

- 合法性校验
- 权限校验
- 版本校验
- 资源校验
- 冲突处理
- 写入动作日志
- 继续推进模拟

### 7.4 实时与半实时折中

建议不要做纯实时无停顿战斗，而采用离散决策窗口：

- 后端固定 tick 推进，例如每秒 10 或 20 tick
- 每经过固定窗口或关键事件时，生成新的决策点
- agent 在限定时长内返回动作
- 超时则使用默认策略或空动作推进

这样可以降低网络和调用抖动对比赛公平性的影响。

## 8. 难度设计与标定

### 8.1 难度目标

四档难度目标如下：

- 简单：50%
- 普通：30%
- 困难：10%
- 地狱：5%

### 8.2 标定方法

不能靠主观感觉定义难度，必须采用标准评测集标定。

建议准备 5 类官方基线 agent：

- 随机策略 agent
- 贪心策略 agent
- 规则驱动 agent
- 轻量搜索 agent
- 官方强基线 agent

然后在固定种子池上反复回归测试，调节数值和机制，直到各档难度在基线评测集上逼近目标通关率。

### 8.3 难度旋钮

四档难度不能只是数值膨胀，建议共同调节以下维度：

- 初始资源
- 商店价格
- 刷新成本
- 稀有遗物出现率
- 精英词缀数量
- Boss 阶段数
- 地图机制复杂度
- 未来波次可预览数量
- 负面事件出现权重
- 拆塔与转型成本
- 敌人抗性成长曲线

### 8.4 四档难度定义

简单：

- 容错率高
- 初始资源较充裕
- 可预览更多未来信息
- Boss 机制较简单

普通：

- 开始要求经济与路线权衡
- 错误决策的恢复成本提升
- 敌人开始出现轻度反构筑机制

困难：

- 必须进行构筑转型
- 事件和商店选择的重要性显著提升
- Boss 会明确针对流派弱点出牌

地狱：

- 几乎要求全局规划无明显失误
- 中后期每一步机会成本极高
- 反构筑机制更强
- 时间优化与稳定性同等重要

## 9. 解锁机制与赛季天梯

### 9.1 逐级解锁

难度按顺序解锁：

- 通关简单后解锁普通
- 通关普通后解锁困难
- 通关困难后解锁地狱

只有通关上一档官方种子池后，才有资格进入下一档。

### 9.2 地狱天梯

地狱难度采用赛季制。

每个赛季发布固定数量的官方种子，例如 5 个。只有全部通关后，才进入地狱排行榜。排行榜优先按总通关时间排序。

建议赛季成绩定义为：

T_season = T1 + T2 + T3 + T4 + T5

推荐排名规则：

- 第一优先级：是否全部通关
- 第二优先级：总通关时间
- 第三优先级：总主堡受损量
- 第四优先级：总动作数

这样可以降低单一特化策略刷单种子的价值。

### 9.3 时间公平性建议

如果按 agent 通关时间排名，最稳妥的方式是官方统一托管运行 agent，而不是直接依赖参赛者自己的公网服务。

更公平的两种方式如下：

- 参赛者提交容器镜像或脚本，平台统一运行
- 如果必须使用远程 A2A 调用，只统计服务端记录的净决策耗时，并限制部署区域

## 10. 数据库设计

建议在 Supabase 中建立以下核心表：

- users
- teams
- agents
- agent_versions
- difficulty_unlocks
- seasons
- seed_pools
- runs
- run_snapshots
- run_actions
- run_events
- run_results
- leaderboards
- replay_assets

### 10.1 表职责概述

users：

- 用户账号信息
- 身份认证关联

teams：

- 队伍信息
- 队伍成员关系

agents：

- agent 基础资料
- 可见性和状态

agent_versions：

- 不同版本 agent 的元数据
- 部署地址或镜像引用
- 评测使用版本标记

difficulty_unlocks：

- 用户或队伍各档难度解锁状态

seasons：

- 赛季信息
- 起止时间
- 当前是否激活

seed_pools：

- 每档难度或每赛季对应的官方种子集合

runs：

- 单局运行记录
- 关联 difficulty、seed、agent_version、状态

run_snapshots：

- 关键帧快照
- 用于回放快速跳转

run_actions：

- agent 动作日志

run_events：

- 战斗结算事件流

run_results：

- 通关结果、耗时、得分、伤害统计、失败原因

leaderboards：

- 不同维度排行榜结果

replay_assets：

- 回放文件、压缩日志、可视化资源索引

## 11. 回放系统设计

回放必须是一等公民，因为它同时承担以下职责：

- 调试 agent 决策
- 定位规则 bug
- 做赛事仲裁
- 验证确定性
- 做内容平衡分析

建议采用“关键快照 + 事件流”的回放格式。

不要每个 tick 全量落库，否则存储成本和读写压力都会很高。正确方式是：

- 每隔若干关键节点保存快照
- 中间过程记录压缩事件流
- 回放时由最近快照开始重演事件

## 12. 可扩展后端设计

### 12.1 API 网关职责

- 身份认证
- 请求限流
- 参数校验
- agent 接入协议统一
- run 创建和状态查询

### 12.2 调度器职责

- 按难度和赛季分配评测队列
- 控制并发数
- 将 run 分配给空闲 worker
- 做失败重试和超时回收

### 12.3 模拟器职责

- 初始化地图、敌人、事件和商店
- 根据种子生成随机内容
- 推进 tick
- 处理攻击、移动、技能、状态、路径、结算
- 处理 action 提交
- 输出关键事件与统计结果

### 12.4 扩容思路

当 run 数量增长后，可以通过以下方式横向扩展：

- 增加 worker 实例
- 将不同难度拆分到不同队列
- 将地狱赛季榜与普通试玩分离
- 对回放生成和统计分析做异步处理

## 13. 确定性与防作弊

这是比赛系统的根基，必须优先保证。

### 13.1 确定性要求

同一版本规则、同一随机种子、同一动作序列，必须生成完全一致的结果。

需要重点注意：

- 随机数生成器统一实现
- 浮点误差控制，尽可能使用定点数或统一精度策略
- 事件执行顺序完全固定
- 并发逻辑不能影响结算顺序

### 13.2 防作弊要求

- agent 不得直接提交状态修改
- 所有 action 必须通过服务端校验
- 对局版本、配置版本、种子版本必须严格绑定
- 榜单成绩必须由官方评测流程产出

## 14. MVP 范围建议

为了尽快形成参赛可用版本，建议第一阶段只做最小可用产品。

MVP 建议范围如下：

- 1 张地图模板
- 4 种基础塔
- 12 种敌人
- 2 个 Boss
- 20 个遗物
- 10 个事件
- 4 档难度
- 完整 agent 接口
- 完整回放链路
- 完整地狱赛季榜

第一阶段不要急着扩内容量，而是先把这些基础能力做稳：

- 服务端权威模拟
- 回放可复现
- agent 接入稳定
- 评测队列可靠
- 排行榜规则跑通

## 15. 推荐开发阶段划分

### 阶段一：规则原型

- 实现地图、敌人、塔、防御结算
- 实现固定种子生成
- 跑通本地单局模拟

### 阶段二：agent 接入

- 定义 observation 和 action 契约
- 支持创建 run 和提交动作
- 做基础超时与回退策略

### 阶段三：Supabase 接入

- 接入用户、队伍、agent 版本
- 持久化 runs、results、leaderboards
- 加入回放文件存储

### 阶段四：难度与内容调参

- 建立基线 agent
- 大规模跑种子池回归
- 调整四档难度通关率

### 阶段五：赛季化与评测化

- 引入官方种子池
- 启动地狱赛季榜
- 固化成绩计算流程

## 16. 结论

这套项目方案的核心竞争力不在于画面，而在于以下四点：

- 规则足够深，适合 agent 参与
- 服务端权威，适合做正式竞赛
- 回放与确定性强，便于调试和仲裁
- 难度与赛季体系清晰，适合长期迭代

对于 A2A 参赛项目而言，这是一条非常务实的路线。它兼顾了实现可行性、比赛公平性、agent 可玩性和后续扩展空间。

- 系统设计文档：模块图、API 契约、数据库 DDL、服务边界
- 游戏规则手册：塔、敌人、Boss、遗物、事件、四档难度的具体数值框架
